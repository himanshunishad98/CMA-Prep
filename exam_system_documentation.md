# CMAPrep Pro - Advanced Exam System Documentation

This document explains the technical details, workflows, and setup instructions for the Advanced Exam Functionality implemented in the CMAPrep Pro application.

---

## 1. Important First Step: Deploying Google Apps Script
Because we added advanced backend capabilities (Level Progression, Test Attempts Data, and Analytics fetching), you MUST update your Google Apps Script deployment.

1. Open your Google Apps Script editor.
2. Ensure the newly updated `Code.gs` is fully pasted and saved.
3. Click **Deploy** -> **Manage Deployments**.
4. Click the pencil icon (Edit) on your current deployment.
5. Under **Version**, select **New version**.
6. Click **Deploy**.
   
*(If you do not create a New Version, the web app will continue running the old code and the new features will throw errors).*

When a student submits their first test, the system will automatically create two new sheets in your Google Sheet file:
- `Test_Attempts_Data`: Stores every detail about the test attempt.
- `Level_Progress`: Tracks the student's email and their current unlocked level.

---

## 2. How the Level Progression Works
The platform now uses a dynamic Level unlocking mechanism based on the user's score percentage on any given test.

**The Rules:**
- Level 1: Unlocked by default upon registration.
- Level 2: ≥ 50% score
- Level 3: ≥ 55% score
- Level 4: ≥ 60% score
- Level 5: ≥ 65% score
- Level 6: ≥ 70% score
- Level 7: ≥ 75% score
- Level 8: ≥ 80% score
- Level 9: ≥ 85% score
- Level 10: ≥ 90% score

**The Workflow:**
1. When a test is submitted, `script.js` calculates the total score.
2. `auth-guard.js` sends the payload to `Code.gs`.
3. `Code.gs` calculates the percentage. If the percentage clears a new threshold (and is higher than their current unlocked level), it updates their record in the `Level_Progress` sheet and returns the `newLevel` back to the frontend.
4. If a student tries to click "Attempt Test" on a test marked as Level 4, but they are only on Level 2, `auth-guard.js` will block them with a popup.

## 3. Anti-Cheating & Violation Control System
To maintain the integrity of the test, strict visibility controls are running in the background.

**Monitored Events:**
- `visibilitychange`: Triggered when the user switches to a different browser tab.
- `window.blur`: Triggered when the user clicks completely outside the browser window (e.g., minimizing it or clicking a different desktop app).

**Violation Ladder:**
1. **First Violation:** An alert pops up warning the student that tab-switching is not allowed, and their next violation will result in a forced submission.
2. **Second Violation:** The test immediately halts, saves the current progress, and forcefully submits to the backend. The submission is flagged as `submissionType: 'violation submit'`.
3. **Permanent Lockout:** If a student tries to manually re-open the exact same test from the dashboard after a violation submit, `localStorage` identifies the block flag (`cmaTestBlocked_[testId]`) and redirects them away permanently.

---

## 4. Auto-Save and Session Persistence
Loss of internet connection or accidental page refreshes will no longer destroy a student's progress.

**The Workflow:**
- A state object (containing selected answers, time left, violations, and marked-for-review questions) is continuously stringified and saved to the browser's `sessionStorage`.
- The save function triggers automatically in three scenarios:
  1. Every 5 seconds via the timer loop.
  2. Every time an answer is clicked.
  3. Every time a question is marked for review.
- If the page refreshes, the `initTestPortal` function detects the existing session token, pulls the exact timer and answers, and resumes the test seamlessly.

---

## 5. Per-Question Time Tracking
Along with total time, the system tracks the granular time spent agonizing over individual questions.

**The Workflow:**
- A `questionStartTime` variable records `Date.now()` the moment a question is rendered.
- When the user clicks "Next", "Previous", or a sidebar grid bubble, the system calculates `Date.now() - questionStartTime` and adds those seconds to a mapping object (`perQuestionTime`).
- On final submission, this data looks like: `{ "0": 45, "1": 12, "2": 120 }` (where the key is the question index and the value is seconds). This is sent to the backend and stored in the Google Sheet for deep analytics.

---

## 6. Admin Panel Analytics
The Admin portal (`admin.html`) was updated to read all of the advanced data generated by the students.

**The Workflow:**
- Clicking the "Refresh Data" button in the Advanced Analytics table triggers `action: 'getAnalytics'` with a `role: 'admin'` payload.
- The Apps Script reads every row in `Test_Attempts_Data` and returns it.
- The Admin UI visually tags the attempts. If an attempt has a `violations` count greater than 0, it renders a bold red "X Violations" tag and shows if it was a Forced submission.

## 7. How to Test It Locally
To ensure everything is working:

1. **Test Anti-Cheating:** Start a test. Click open a new tab, then go back. You should see a warning. Do it again, and the test should auto-submit and redirect you to the dashboard.
2. **Test Auto-Save:** Start a test. Select an option for Question 1. Wait 5 seconds. Hit Refresh (`F5`) on your browser. Your timer should not restart, and Question 1 should still be locked in with your answer.
3. **Test Level Progression:** Ensure the JSON test you upload has a total marks configuration that allows you to get 90%. Submit the test with 100% correct answers. Check your dashboard profile—it should instantly say "Unlocked Level 10". 
4. **Test Admin Visibility:** Go to `admin.html` and click "Refresh Data". You should see your recent attempt logged, alongside any cheating flags you triggered.
